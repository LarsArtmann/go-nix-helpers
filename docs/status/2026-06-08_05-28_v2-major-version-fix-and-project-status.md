## a) FULLY DONE ✅

### 1. `mkPreparedSource` — Core helper

- Copies private flake-input deps into `_local_deps/`
- Injects `replace` directives into `go.mod`
- Supports `subModules` (repos with independent sub-module `go.mod` files)
- Supports `requireDeps` (manual require injection for sub-modules not yet in `go.mod`)
- Strips stale local `replace` directives pointing to `/home/...`
- Normalizes sub-module pseudo-versions via `subModuleVersion`

### 2. `/v2`/`/v3` major version suffix handling — JUST FIXED

- **Problem**: `basename = lib.last (lib.splitString "/" path)` extracted `"v2"` instead of `"go-filewatcher"` for paths like `github.com/larsartmann/go-filewatcher/v2`
- **Impact**: Collisions when multiple `/v2` deps from different repos coexist (`_local_deps/v2` overwrites), meaningless directory names, broken `postPatchExtra` that hardcodes expected names
- **Fix**: Added `repoName` helper that strips `/vN` suffixes before extracting basename
- **Verified**: `nix eval` confirmed correct output for all edge cases (go-output, go-filewatcher/v2, gogenfilter/v3, template-LICENSE/types, go-output/v10)
- **Downstream blast radius**: **ZERO** — no consumer hardcodes `_local_deps/<version-suffix>` in `postPatchExtra`

### 3. Template — `go-flake-parts`

- Complete flake template with flake-parts, treefmt-nix, systems
- Commented-out `mkPreparedSource` usage with `/v2` and `subModules` examples
- CI devShell, lint/test apps, format checks, overlay pattern

### 4. README documentation

### 1. Downstream consumers — Need `vendorHash` updates

Two projects use versioned deps via mkPreparedSource and will need `vendorHash` recalculated after this fix:

| Project                          | Versioned Deps                        | Risk                                                   |
| -------------------------------- | ------------------------------------- | ------------------------------------------------------ |
| `projects-management-automation` | `go-filewatcher/v2`, `gogenfilter/v3` | Low — `_local_deps` names change but no hardcoded refs |
| `go-structure-linter`            | `gogenfilter/v3`                      | Low — `_local_deps` names change but no hardcoded refs |

These projects previously had `_local_deps/v2`, `_local_deps/v3` — now they'll get `_local_deps/go-filewatcher`, `_local_deps/gogenfilter`. The `replace` directives in `go.mod` are auto-generated by mkPreparedSource, so they'll update correctly. Only `vendorHash` needs recalculation.

### 2. `projects-management-automation` — Manual `postPatch` workarounds

PMA manually copies `cmdguard` into `_local_deps/cmdguard` in its own `postPatch` block (outside mkPreparedSource). This is the kind of manual workaround that mkPreparedSource should handle. With the `/v2` fix, this can now be moved into `deps` directly:

```nix
"github.com/larsartmann/cmdguard/v2" = cmdguard;
```

## d) TOTALLY FUCKED UP 💥

### Nothing is fucked up.

The `/v2` bug existed but was latent — it only manifested when a project used `/v2`/`/v3` deps AND relied on auto-generated `_local_deps/` names. The existing consumers with versioned deps (`pma`, `go-structure-linter`) happened to work because:

1. `gogenfilter/v3` → `_local_deps/v3` — worked in isolation (single `/v3` dep)
2. `go-filewatcher/v2` → `_local_deps/v2` — worked in isolation (single `/v2` dep)
3. No project had **multiple** same-version deps (e.g., two different `/v2` repos) that would have collided

So the bug was real but never triggered in practice. Fixed before it caused damage.

## e) WHAT WE SHOULD IMPROVE 📈

### 1. Test infrastructure

Add a `flake.nix` with Nix checks that verify `repoName` output for various paths. This would have caught the `/v2` bug at introduction.

### 2. Reduce downstream manual work

`projects-management-automation` manually copies `cmdguard` into `_local_deps/`. With the `/v2` fix, this can move into `deps`. Audit all consumers for similar workarounds.

### 3. Error handling for ambiguous paths

`repoName` treats `template-LICENSE/types` correctly (extracts `"types"`, not a version suffix). But what about paths like `github.com/foo/bar/v2/baz`? Currently extracts `"bar"`. This is correct for Go module paths (version suffix is always last segment), but worth documenting.

### 4. `subModuleVersionNormalize` is fragile

The sed command `s|${depPath}/${sub} v0\.0\.0-[^ ]*|...|g` assumes a specific pseudo-version format. If Go changes the format, it breaks silently.

### 5. Template auto-generation

The `templates/go-flake-parts/` template is manually maintained. Consider generating it from the main `mkPreparedSource.nix` comments to avoid drift.

## g) Top #1 Question I Cannot Figure Out Myself 🤔

**Should `repoName` produce a unique name when two deps have the same repo name but different owners?**

Example:

```nix
deps = {
  "github.com/larsartmann/go-output" = go-output;
  "github.com/otheruser/go-output" = other-go-output;
};
```

Currently `repoName` would extract `"go-output"` for both, causing a collision in `_local_deps/`. Is this a realistic scenario? If yes, we need to include the owner/namespace in the directory name (e.g., `_local_deps/larsartmann-go-output`).
