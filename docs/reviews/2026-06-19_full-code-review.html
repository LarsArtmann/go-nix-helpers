01Fixed on the spot

These are objective defects confirmed by execution. Each was fixed and re-verified.

Critical · Fixed

Template references an unregistered nixpkgs maintainer

templates/go-flake-parts/flake.nix:112

maintainers = [ maintainers.larsartmann ]; throws "attribute 'maintainers.larsartmann' missing" because larsartmann is not in nixpkgs' maintainer-list.nix. Verified: nix eval lib.maintainers ? larsartmann → false. This breaks nix build and nix flake check for every new project generated from this template — the highest-blast-radius bug in the repo.

Fix applied Removed the maintainers line with a comment explaining handles must be registered in nixpkgs before referencing them. Register Lars in nixpkgs to re-enable.
High · Fixed

Private-dep validation hardcoded to one GitHub org

mkPreparedSource.nix:269 (validateScript)

The validation grep was hardcoded to github\.com/[Ll]ars[Aa]rtmann/. A library named generically go-nix-helpers silently passes any non-LarsArtmann private dep through to the cryptic SSH error it was designed to prevent. This is a core extensibility flaw — the validation only protects one org.

Fix applied Added a privateDepPattern parameter (ERE regex) defaulting to the existing LarsArtmann pattern — fully backwards-compatible. Verified end-to-end: a custom gitlab\.com/mycompany/ pattern correctly triggers validation, the default does not. Documented in README's parameter table.
High · Fixed

Integration test verify never actually passed

test.nix (verify derivation)

Two compounding faults: (1) the elif grep "builder failed" branch matched any build failure, so a syntax error would count as a PASS — a false-positive test; (2) validationTest (a deliberately-failing derivation) was referenced as a Nix dependency of verify, and a failing derivation can never be in the closure of a passing one, so nix-build -A verify always failed with "1 dependency failed". The suite documented as "working" had never passed end-to-end.

Fix applied Restructured: verify now tests only success paths (auto-discovery + explicit) and passes. The negative case moved to verifyValidation — a writeShellApplication run outside the sandbox (the sandbox lacks nix-store), asserting the exact validation message. Both verified green.
Medium · Fixed

nix-lint.sh grep produces garbage, breaking the comparison

scripts/nix-lint.sh:91

checks_count=$(grep -cP ... || echo "0") — grep -c prints 0 AND exits 1 on no match, so || echo 0 appends a second 0, yielding the string 0\n0. Verified: the variable literally contained a newline. [ "$checks_count" -gt 1 ] then errors on "integer expression expected". The duplicate-checks lint rule was silently broken.

Fix applied Rewrote to assign inside the if conditional so the exit code is consumed directly, avoiding the double-echo.
Medium · Fixed

docs/ci-workflow.yml:30

Fix applied Removed || true with a comment noting when to re-enable leniency.
Medium · Fixed

scripts/dashboard.sh:33,47,58

Three faults: (1) hardcoded /home/lars/projects — not portable, not configurable; (2) warned only on go_1_25 while the standard is go_1_26 (and the check is backwards — it never flags older versions); (3) raw ${error} interpolated into JSON without escaping, producing invalid JSON on any error containing a quote.

02Open findings (need a decision)

These were not auto-fixed because they involve API/tradeoff decisions or larger work. Ranked by impact.

High · Open

No flake.nix — the repo cannot dogfood itself

project root

The #1 gap, flagged in both prior status reports. A Nix library that ships a flake-parts + treefmt-nix gold-standard template does not apply that standard to itself. Consequences: no nix fmt on its own .nix files, no CI gate on test.nix, no nix flake check. The "imported as flake = false" rationale is not a blocker — a repo can be both a proper flake (exposing lib.mkPreparedSource) and importable via raw path. This is the highest-leverage improvement available.

Medium · Open

goPkg parameter is dead weight

mkPreparedSource.nix:68,297

goPkg is a required parameter and added to nativeBuildInputs, but the derivation has dontBuild = true and the builder only runs sed/cp/mkdir/echo/awk/grep — never go. The Go toolchain is pulled into the sandbox for nothing, inflating the closure and (worse) signalling to consumers that pinning the Go version here matters. It does not. Removing it is a breaking API change for all consumers, so left for an explicit decision (deprecate, default it, or drop).

Medium · Open

Parallel code paths for explicit vs auto sub-modules

mkPreparedSource.nix:182-247

subModuleReplaceExplicit/subModuleReplaceAuto and subModuleVersionNormalize/autoVersionNormalize are near-identical pairs doing the same job on different shapes. This is a split brain waiting to drift — the /vN fix had to be applied in both. Both should reduce to a single function over a unified [{modulePath, localDir}] list (the allDiscovered shape already exists; explicit entries should map into it).

Medium · Open

repoName collision for same-name different-owner deps

mkPreparedSource.nix:99-101

repoName extracts only the last path segment, so larsartmann/go-output and otheruser/go-output both produce _local_deps/go-output — a silent overwrite. Not triggered today but architecturally unsound. Fix: namespace by owner (e.g. _local_deps/larsartmann-go-output).

Medium · Open

Pseudo-version normalization may be cargo-culted

mkPreparedSource.nix:226-247

subModuleVersionNormalize rewrites v0.0.0-20260101000000-abc123 → v0.0.0 so replace directives "match". But Go ignores the require version for path-replaced modules, so the sed may be unnecessary. The sed also assumes a specific pseudo-version format and breaks silently if Go changes it. Worth a controlled test: remove it and see if a real build still resolves.

Medium · Open

generate-flake.sh: fragile sed, hardcoded paths, surprising push default

scripts/generate-flake.sh:28,35,53-57,76

Hardcoded /home/lars/projects; the --templ sed injections are line-number-dependent and the line-57 pattern doesn't match the template's multi-line formatter block (likely broken); sed "s/REPLACE_ME/$PROJECT/g" breaks if PROJECT contains /; and the script defaults to interactively pushing to GitHub, which is surprising for a generator. Defaults should be non-destructive.

Low · Open

No AGENTS.md / CHANGELOG.md / FEATURES.md / TODO_LIST.md

project root

Per project documentation conventions, every project should have an AGENTS.md for AI-session continuity. The two status reports under docs/status/ are excellent point-in-time snapshots but do not substitute for enduring context.

Low · Open

hasRequires can emit duplicate require lines

mkPreparedSource.nix:222,311

requireDeps appends a fresh require (...) block without checking whether a module is already required. A duplicate require makes go mod tidy complain. Should dedup against existing requires.

Low · Open

Undocumented postPatchExtra ordering

mkPreparedSource.nix:306

postPatchExtra runs before the replace (...) block is appended. A consumer whose postPatchExtra needs to read the generated replaces will silently see a stale go.mod. Document the order, or move it to the end.

Low · Open

mockDep uses an anti-pattern (writes $out in unpackPhase)

test.nix:18-40

mockDep writes to $out inside unpackPhase (which should populate the build dir), then installPhase copies . over it. Confusing and incidental. Replace with pkgs.runCommand or do all file creation in installPhase.

03Core helper deep-dive

mkPreparedSource.nix (334 lines) is the heart of the project. Its core idea is sound and well-executed: copy flake-input deps into _local_deps/ and inject replace directives so the Go toolchain resolves private modules locally without network/SSH. The auto-discovery design (scanning for subdirectory go.mod files) is genuinely good — it eliminates manual subModules maintenance and is the kind of "make the right thing automatic" decision that ages well.

mkPreparedSource ├── auto-discovery (scan go.mod) ├── replace injection ├── /vN suffix handling ├── clear validation errors ├── duplicated explicit/auto paths ├── dead goPkg param └── org-hardcoded validation (now parameterized)

The main architectural debt is the four near-duplicate shell-generation blocks. Unifying explicit and auto sub-module handling behind one function would halve the surface area and make the next /vN-class bug impossible to introduce in only one branch.

♦Strengths

+
Tight, single-purpose scope One job, done well. No scope creep. Easy to reason about and trust.
+
Auto-discovery eliminates a whole class of maintenance Scanning dep sources for go.mod means new sub-modules are picked up on nix flake update with zero config. Excellent DX.
+
Validation turns a cryptic error into an actionable one The "could not read Username" SSH error is notoriously opaque. The build-time check listing missing deps is exactly the right usability investment.
+
Surrounding ecosystem (template, lint, dashboard) The repo doesn't just ship a helper — it ships the scaffolding to use it consistently across 7+ projects. The lint rules codify hard-won lessons.
+
Honest, detailed status reports The docs/status/ reports are candid about what's done, partially done, and "totally fucked up". That honesty is rare and valuable.

04Recommended next steps (Pareto)

Ordered by impact-to-effort ratio. The first two unlock the most value.

1

Add a flake.nix that self-hosts the repo

Expose lib.mkPreparedSource, add treefmt-nix (nixfmt), wire checks to test.nix. Keeps raw-path import working. Single highest-leverage change — dogfoods the standard the repo preaches.

2

Unify explicit + auto sub-module code paths

Map explicit subModules entries into the same [{modulePath, localDir}] shape as allDiscovered, then run one replace-generator and one normalizer. Halves the bug surface.

3

Decide on goPkg and the pseudo-version sed

Confirm goPkg is unused (it is) and deprecate/drop it. Test whether subModuleVersionNormalize is actually needed by building a real consumer without it.

4

Namespace repoName by owner; add AGENTS.md

Eliminates the same-name collision and gives every AI session the project context it currently lacks.
